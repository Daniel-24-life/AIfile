# AI 使用手册

副标题：从文章阅读到工作能力

### 1. 我为什么写这份手册

这份手册记录我最近阅读 AI 工程、Agent、Harness、Loop Engineering、AI Native Coding 等材料后形成的使用方法。

我最关心的问题很实际：

```text
读完这些 AI 文章之后，我下次做真实任务时会有什么不同？
```

目前沉淀出的答案是：我要把 AI 放进一个更清楚的工作系统里。这个系统包含目标、边界、上下文、工具、状态、验收和复盘。输出只是其中一环，真正决定质量的是整条工作链路。

### 2. 我的核心公式

```text
Agent = Model + Harness
```

模型负责推理和生成。Harness 负责让模型在真实任务中稳定工作，包括：

- 上下文
- 工具
- 权限
- 文件状态
- 检查清单
- 验收证据
- 复盘和知识回流

以后我使用 AI 时，会先设计任务环境，再开始要求产出。

### 3. 任务启动模板

重要任务开始前，我用这个模板描述需求：

```markdown
目标：

背景：

输入材料：
- 

成功标准：
- 

允许范围：
- 

禁止范围：
- 

验收方式：
- 

需要沉淀：
- 规则 / 模板 / checklist / Skill 候选 / 复盘
```

如果任务复杂，我会先让 AI 复述目标、边界和验收方式。确认理解一致后，再进入执行。

### 4. 任务分级

不同任务需要不同强度的工作系统。

| 任务类型 | 例子 | 使用方式 |
| --- | --- | --- |
| 简单问答 | 概念解释、翻译、短句改写 | 直接问，快速确认 |
| 资料消化 | 文章、PDF、报告 | 结构化提炼，再转行动清单 |
| 工作产出 | 文章、方案、PPT 大纲、作品说明 | 先定受众、场景和成功标准 |
| 代码/文件修改 | 修 bug、整理项目、生成工具 | 读地形、定边界、改动、验证 |
| 长任务 | 跨文件、跨轮次、跨天项目 | 状态落盘、阶段验收、handoff |
| 风险任务 | 删除、批量修改、联网、发布 | 人工确认、路径核对、回滚方案 |

我的判断口令：

```text
不可逆、跨文件、对外部有影响、需要多轮完成 = 加强 Harness。
```

### 5. AI Native Coding 的使用方法

我从 AI Native Coding 相关材料里吸收的核心判断是：

```text
代码生成会越来越快，真正昂贵的是目标、边界、证据、风险和验收。
```

所以代码任务按七步走：

1. 读地形：先理解项目结构、相关文件、测试入口。
2. 切任务包：把任务切到小到可检查、大到可自治。
3. 写 spec：把目标、边界、验收标准落成外部真相源。
4. 放手执行：允许 AI 自主读代码、改代码、跑测试、修 bug。
5. checkpoint 管控：人在关键节点决定放行、追问、加约束、回炉或阻止。
6. 验收证据：看 diff、测试、构建、日志、产物路径。
7. 必要时重启：长对话混入太多旧假设后，用 spec + handoff 开新任务。

### 6. Loop Engineering 的使用方法

Loop Engineering 帮我区分两件事：

```text
Harness 解决一次任务怎么做。
Loop 解决任务如何持续转起来。
```

我不会一开始就追求自动化。一个任务值得 Loop 化，需要通过四个问题：

```text
1. 这个任务会重复发生吗？
2. 验证能自动或半自动完成吗？
3. 每轮运行成本可承受吗？
4. AI 是否有足够工具和权限完成闭环？
```

如果条件不足，就先采用：

```text
半自动陪跑 + 人工 checkpoint
```

如果条件成熟，再考虑 Connectors、Automations、Skills、Worktrees、Sub Agents 和 State files。

### 7. 验收证据链

AI 的回答再流畅，也需要证据支撑。

我把验收分成五层：

| 层级 | 证据 |
| --- | --- |
| L1 | AI 自述做了什么 |
| L2 | 文件产物存在且路径明确 |
| L3 | 测试、构建、lint、渲染或扫描通过 |
| L4 | 独立 review 或第二视角检查 |
| L5 | 真实使用、监控、灰度、回滚或用户反馈 |

普通任务到 L2/L3 通常够用。涉及线上、数据、权限、安全时，要提高到 L4/L5。

### 8. 状态落盘

长任务需要外部状态。

我会把这些内容写进文件，减少对聊天上下文的依赖：

- task spec
- current status
- decision record
- handoff
- verification report
- reusable template
- next action

这能减少长对话里的旧假设、临时判断和上下文污染。

### 9. K/S/T：阅读成果的整理方式

为了把阅读真正变成能力，我把知识拆成三层：

| 层 | 含义 | 产物 |
| --- | --- | --- |
| K: Knowledge | 事实、概念、材料理解 | 摘要、概念图、知识卡 |
| S: Standards | 角色、边界、规范、验收 | checklist、禁止事项、质量门 |
| T: Tactics | 流程、卡点、复盘、回流 | SOP、任务模板、handoff |

读文章时，我会问：

```text
这条收获应该进入 K、S，还是 T？
```

这样整理后，每条阅读收获都有去处，也更容易进入下一次真实任务。

### 10. 这轮阅读带来的能力变化

| 阅读主题 | 我吸收后的能力变化 |
| --- | --- |
| AI Native Coding | 从关注实现速度，转向关注目标、边界、checkpoint 和证据链。 |
| Loop Engineering | 开始判断哪些任务值得做成持续闭环。 |
| Harness / 脚手架 | 更关注上下文、工具、状态、权限和验收环境。 |
| Multi-Agent Harness | 更重视角色拆分、独立审查和状态恢复。 |
| 模型可解释性 | 不把模型自述当证据，更多依赖外部行为和测试。 |
| 超级个体 / 超级组织 | 把个人经验整理成 K/S/T 资产。 |
| 长程 AI 开发流程 | 更重视提案、审查、执行、验证的角色分离。 |

### 11. 我的 AI 能力指标

| 指标 | 好的迹象 |
| --- | --- |
| 任务表达能力 | 能说清目标、成功标准和边界 |
| 上下文控制能力 | 给 AI 的材料更少但更准 |
| 验收能力 | 能判断产物是否可用 |
| 复盘能力 | AI 犯错后能沉淀规则 |
| 系统化能力 | 常见任务逐渐有模板、清单、状态文件和 Skill 候选 |

### 12. 下一步

这份手册需要进入真实任务验证。

我会用三类任务检验它：

1. 资料消化：把长文变成框架、启发和行动清单。
2. 工作产出：把主题变成文章、方案或作品说明。
3. Codex 操作：让 AI 在有限范围内完成文件整理、代码修改或 GitHub 作品上架。

每次任务结束后，我会问：

```text
这次有没有沉淀出一条可复用规则、模板、检查项或 Skill 候选？
```
